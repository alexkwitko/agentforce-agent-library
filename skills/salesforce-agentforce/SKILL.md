# Salesforce Agentforce + Data Cloud + Messaging-for-Web Playbook

A practical, **product-agnostic** playbook for building and operating an Agentforce solution on any org, in any industry (commerce, financial services, healthcare, B2B, etc.). The guidance and gotchas are general; concrete examples are drawn from real builds but the patterns apply to any domain ‚Äî substitute your own objects (an `Order` example maps equally to a Claim, Application, Policy, or Work Order). Follow the DX-first ordering below; only fall to the browser/UI when there is genuinely no API/metadata/CLI path.

## The #0 rule: Official-first ‚Äî research the supported Salesforce way; never Frankenstein a custom workaround

Before building ANYTHING, find the **official, supported Salesforce mechanism** for the requirement and use it. Research it actively ‚Äî official Salesforce docs (help.salesforce.com, developer.salesforce.com), the `context7` MCP for product docs, and web search ‚Äî *based on the actual requirement*, not from memory or assumption. Standard objects and native product features almost always already cover the need.

**Do NOT hand-roll a custom object + Apex to mimic a capability the platform already provides** ‚Äî not to satisfy a design quickly, not to stay "headless," not to dodge a setup/UI step, not because it's faster to write. That is a **Frankenstein solution**: it demos fine but duplicates a platform feature, bloats storage, drifts from the product's roadmap, reimplements (badly) something Salesforce already does correctly, and becomes permanent tech debt the next person must untangle.

**Real anti-example from this playbook's own history:** a custom `Web_Event__c` object + Apex `IdentityStitchService` was built to do device‚Üíperson identity stitching and store browsing ‚Äî when the **official** answer is the **Data Cloud Web SDK + native Identity Resolution** (deviceId/anonymousId match rules). The custom version "worked," but it mirrored high-volume web events into a 5 MB CRM org (repeated storage outages) and re-implemented identity resolution by hand. The right move was to research and use the native Data Cloud capability from the start.

**Decision rule for every requirement:**
1. Ask *"What is the official Salesforce product / feature / standard object / declarative tool for this?"* ‚Äî then research to confirm; do not guess.
2. Use it in this order: **standard object/native feature ‚Üí declarative (Flow) ‚Üí thin Apex glue invoked by Flow ‚Üí custom object/Apex (last, only with a stated reason no official path exists).**
3. If the official path needs a UI/Setup step (Identity Resolution match rules, a Data Cloud connector, an Einstein Studio model), **do the UI step.** Correct-and-supported beats convenient-and-custom. Never substitute a custom Apex hack just to avoid the official setup.
4. Show the research: when choosing custom, name the official options ruled out and why; when choosing official, name the feature.

"Do it right" = the supported way, even when it's more work or needs a tool you don't currently have ‚Äî not the fastest custom thing that satisfies the demo. See `crm-vs-datacloud-architecture.md` and `standard-objects-and-declarative-first.md`.

## The #1 rule: DX/CLI/API first, UI last

Preference order for every task:
1. **Salesforce DX / `sf` CLI** ‚Äî `sf project deploy/retrieve`, `sf data`, `sf apex run`, `sf agent`.
2. **Connect/REST API via CLI** ‚Äî `sf api request rest "/services/data/vXX.0/..."` (uses the stored session; works even when the raw access token is hidden).
3. **Metadata XML deploy** ‚Äî author the component, deploy it.
4. **Apex / REST** for anything procedural.
5. **Browser/UI automation** ‚Äî last resort, and say so first. A handful of things are genuinely UI-only (listed in `troubleshooting.md`): creating CRM data streams, attaching a User-Verification keyset to a MIAW deployment, setting the Person-Account default record type on a profile.

> In a sandboxed agent environment, `sf org display` REDACTS the access token. Do NOT try to scrape it. Instead use `sf api request rest` / `sf data` / `sf apex run`, which all reuse the CLI's stored session under the hood.

## Fast-path checklists

**Deploy a change:**
```bash
sf project deploy start --metadata ApexClass:MyService --target-org MyOrg
sf project deploy report --use-most-recent   # deploys are ASYNC ‚Äî wait for final "Succeeded"
```
After deploying NEW custom fields, deploy + assign a permission set granting FLS or SOQL will report "No such column".

**Build/ship an agent:**
```bash
sf agent generate agent-spec ... ; sf agent generate authoring-bundle ...
# edit aiAuthoringBundles/<Name>/<Name>.agent
sf agent validate authoring-bundle --api-name <Name>
sf agent publish authoring-bundle --api-name <Name>
sf agent activate --api-name <Name> --version <N>   # ‚öÝÔ∏è publish does NOT activate ‚Äî new version stays Inactive
# verify the version you published is the ACTIVE one before testing:
sf data query --query "SELECT VersionNumber,Status FROM BotVersion WHERE BotDefinition.DeveloperName='<Name>' AND Status='Active'"
```
Agent type (employee vs Service Agent) is **immutable after first publish** ‚Äî get it right the first time (see `agentforce-agents.md`).
> ‚öÝÔ∏è **`sf agent publish` does NOT activate the new version** ‚Äî it's created `Inactive` while the OLD version keeps serving live traffic AND keeps being what `sf agent test`/`run-eval` exercises (they resolve `Status='Active'`). You MUST `sf agent activate --api-name <Name> --version <N>` and then confirm the active version, or you'll "fix" the agent and keep testing the unchanged one. Always verify the active version before drawing ANY conclusion from a test run.

**Run an agent headlessly (from Apex/Flow):**
```apex
AgentInvoker.Response r = AgentInvoker.callAgent('My_Agent_Api_Name', 'user message');
```

**Run a Data Cloud Identity Resolution / Calculated Insight (Connect API):**
```bash
# IR run-now ‚Äî body is {} and the path is /actions/run-now (NOT /actions/run)
sf api request rest "/services/data/v62.0/ssot/identity-resolutions/<id>/actions/run-now" --method POST --body '{}'
# Calculated Insight run
sf api request rest "/services/data/v62.0/ssot/calculated-insights/<apiName>/actions/run" --method POST --body '{}'
# Ad-hoc SQL query against DLM/DLL/CIO
sf api request rest "/services/data/v62.0/ssot/query-sql" --method POST --body '{"sql":"SELECT COUNT(*) FROM UnifiedssotIndividual__dlm"}'
```

**Diagnose a dead chat / mysterious "can't create records":** check storage FIRST (see `troubleshooting.md`).
```bash
sf api request rest "/services/data/v62.0/limits" | jq '.DataStorageMB'
```

## Top lessons learned (read these even if you read nothing else)

00. **Decide CRM-vs-Data-Cloud placement BEFORE building (the costliest architecture choice).** CRM is the transactional system of record ‚Äî only the records an agent/Flow/Apex must *act on* (the serviced Account/Contact, Case, in-flight transactional record). Data Cloud is the analytical/unification layer beside it ‚Äî 360, high-volume events, identity resolution, scores/segments. **Do NOT mirror an entire external system (commerce, ERP, billing, etc.) into CRM to make it agent-readable** ‚Äî it exhausts CRM row storage, forces brittle Apex single-key matching that forks customers when that key (e.g. email) changes, and leaves a stale 360. Instead: stream-ingest into Data Cloud, let Identity Resolution own matching, Data-Cloud-Triggered-Flow a *thin* record into CRM, and have the agent **read the 360 from the unified profile and write transactional actions to the system of record (CRM or the external API)**. See `crm-vs-datacloud-architecture.md`.

0a. **Discover before you design: interview the user, research the industry, confirm the concrete inputs.** Never build from assumptions about the business process, where data lives, or how identity arrives ‚Äî ask, and for anything they can't recite, get the artifact (object/field API name, endpoint URL, screenshot of the setting/UI, sample record). Bring researched best-practice as a *proposal* to confirm, not a guess to ship. The build pipeline is **discover ‚Üí design ‚Üí build ‚Üí deploy ‚Üí verify**. See `discovery-intake.md`.
0. **Design before you prompt: map every process ‚Üí every outcome ‚Üí the action that produces it, plus a defined branch for "can't."** An agent built straight into prose instructions, without a capability map + process/outcome matrix + per-process decision tree, WILL hallucinate ‚Äî it will "trust" an unverified user, narrate actions it never called, and claim refunds/cases that don't exist. Verification is a GATE on a verified variable, never a user's claim ("I'm signed in" ‚âÝ verified). The agent may assert an outcome ONLY if the action returned its success flag this turn; every branch ends in a real result or an honest "I can't" + escalation. See `solution-design-process-outcomes.md`.
0b. **Standard before custom, declarative before code, DX before UI.** Before any custom object/field or Apex class, research whether a STANDARD object already models it (Order/OrderSummary, Case, ReturnOrder, FulfillmentOrder, Shipment, CreditMemo, Account+Contact, Individual/ContactPoint, MessagingSession‚Ä¶) and whether a FLOW can do the automation instead of Apex. Data need: standard object ‚Üí standard + custom fields ‚Üí custom object (last). Automation: Flow ‚Üí thin `@InvocableMethod` Apex called BY a Flow/agent ‚Üí pure Apex (last). Inspect with `sf sobject describe`; confirm real API names from the Object Reference ‚Äî never guess `__c`. See `standard-objects-and-declarative-first.md`.
1. **DX/CLI/API beats the screen** for ~90% of Agentforce + Data Cloud work, including things people assume are UI-only (DLO‚ÜíDMO maps, Calculated Insights, agents, routing flows, smoke tests). Author as metadata + deploy with `sf`; reserve the UI for the handful of genuinely UI-only steps.
2. **Deploys are async** ‚Äî always confirm final "Succeeded" via `sf project deploy report`, not the first component table.
3. **New custom fields are invisible until a permission set grants FLS** ‚Äî deploy + assign a permset every time.
4. **Agent type is immutable** after first publish; employee agents must NOT have `default_agent_user` / `connection messaging:` / `@MessagingSession` vars, and a null `BotDefinition.BotUserId` on an employee agent is a reporting artifact, not a failure.
5. **Data Cloud Identity Resolution silently yields 0 unified profiles when the identity keys are sourced from an EMPTY field** ‚Äî map Individual PK and ContactPointEmail PartyId from a POPULATED field (e.g. `Account_Home__dll.Id__c`), not an empty one like `PersonIndividualId__c`. Verify with `SELECT COUNT(*) FROM Unified...__dlm`.
6. **A Calculated Insight can read a `__dll` (DLO) directly** ‚Äî you don't always need a DLO‚ÜíDMO map for analytics CIs.
7. **CRM-source data streams cannot be created via metadata** ("no MktDataTranObject") ‚Äî that step is UI-gated; everything on top of an existing stream (maps, CIs) is DX-able.
8. **CI SQL has strict rules**: no table aliases (use the full DMO name as qualifier), output aliases must end in `__c`, currency measures must be wrapped in `TRY_CONVERT_CURRENCY(...)`, and a literal target-currency dimension must appear in BOTH SELECT and GROUP BY.
9. **MIAW user verification**: do NOT assume "v2 removes user verification" ‚Äî modern WebV2 exposes `userVerificationAPI`. The real blocker is the served config's `authMode=UnAuth`; until that's flipped (keyset attached to the deployment, often a Support/UI step) `setIdentityToken` is silently ignored.
10. **MIAW `POST /conversation` ‚Üí 412 "Conversation Precondition failed" is usually a FULL data-storage org**, not a config error. When storage is exhausted no records (incl. MessagingSession) can be created, so chat, site provisioning, and backfills all fail at once.
11. **AI Evaluation results from `sf agent test` are a massive silent storage hog and are NOT deletable via Apex/Tooling** (Testing Center UI only). Don't run agent tests on a storage-tight org. Deleted records keep consuming storage until you `Database.emptyRecycleBin`.
12. **Agents are conversational, not event-driven** ‚Äî for self-service automation, fire the agent headlessly from a scheduled job / Flow (via `AgentInvoker`), or do the work in a `without sharing` Apex service. External-webhook-triggered work runs as the locked-down **Site guest user**, so put trusted back-end logic in `without sharing` scheduled sweeps, not guest-context Flows.
13. **Outbound email from a @gmail.com (or any unowned-domain) From address fails Gmail DMARC** even though Apex reports `isSuccess=true`. Real deliverability needs an Org-Wide Email Address on a domain you control with SPF + Salesforce DKIM.
14. **Certify agents with MULTI-TURN conversation tests, repeated to 100% ‚Äî single-turn tests lie.** An agent that passes single-turn `sf agent test run` will still fabricate actions live, because it behaves differently when users drip info across turns (the normal case). Proven: single-turn 100% green while the same return flow failed ~1-in-3 multi-turn runs. For EVERY agent: document all use cases as a matrix, replay each as a dripped multi-turn conversation via `sf agent test run-eval` (JSON: one `create_session` + a `send_message` per turn + `evaluator.*` assertions), run each critical case ‚â•10√ó and require 100% (LLM output is non-deterministic). Never ship a rubric that endorses insecure behavior (e.g. acting on a typed-but-unverified email). See `multi-turn-agent-testing.md`.
15. **"Email limit exceeded" is a sending-METHOD problem, not a provider problem.** The DE ~15/day external-email cap counts only `setToAddresses` sends; emailing a **Lead/Contact/User record via `setTargetObjectId` is cap-EXEMPT** and still sends from your verified OWEA. That's the fix ‚Äî not a new ESP or a bolted-on relay. (Deliverability ‚Äî mail "sends" but doesn't arrive ‚Äî is the separate DMARC/DKIM problem; marketing at scale belongs in Marketing Cloud.) See `email-and-notifications.md`.
16. **Apex agent-action classes: only ONE `@InvocableMethod` per class** (split each action into its own thin wrapper over a shared `static` logic class), inputs/outputs must match the `@InvocableVariable` names, and actions should **return a structured `{success,message}`** (not `void`) so the agent can honestly confirm. And: **you can't edit an Agentforce Builder (Studio) draft headlessly** ‚Äî rebuild it as a DX `aiAuthoringBundle` with the same dev name and `publish`+`activate` to replace it. 17. **Agentforce DECIDES; it must not SEND marketing.** Enterprise pattern: agent *nominates* a person into a **Data Cloud segment**, a **consent-aware activation** publishes only consented individuals, and **Marketing Cloud** sends. Unify consent from every source app into Data Cloud's standard Consent model, reconcile to ONE effective consent (latest-wins/opt-out-sticky), and **enforce it at the activation, not in app code.** See `consent-and-marketing-activation.md`.
18. **MIAW identity is bound at conversation creation and never re-checked per turn** ‚Äî causing a logout PII leak (signed-out user still gets PII) and the mirror over-block (mid-chat login not recognized). Fix the client bridge to **end the conversation on any identity change** (`utilAPI.clearSession({shouldEndSession:true})` + an identity fingerprint) and render identity server-side; the durable fix is **JWT token expiry**. Verified vs guest is **all-or-nothing per deployment** (guests need a separate deployment), and a **cached page/stale nonce** makes a signed-in user look like a guest. See `messaging-web.md`.
19. **Gate privileged actions STRUCTURALLY, not in prose.** Put verification/eligibility in a no-tool "gate" subagent and remove the privileged tools from the pre-auth topic ‚Äî the planner then physically can't act early. Source every authorization fact from a `linked` channel/record variable (a typed/looked-up value is never proof); require an **opaque OTP proof token** re-verified by Apex on every gated call. See `agent-orchestration-and-memory.md`.
20. **Cross-agent state lives in CRM, not the conversation.** Platform memory doesn't survive a handoff to a different agent or a later headless run ‚Äî model a per-entity journey object + append-only interaction log + idempotent "done sets" (check-then-mark), and make get-or-create on a unique key race-safe (catch `DUPLICATE_VALUE`, re-SELECT). Make ONE shared-context action the mandatory first call in every subagent. See `agent-orchestration-and-memory.md`.

22. **Know the whole Data Cloud toolbox, pick by the requirement:** ingest by source (CRM connector / Ingestion API streaming-or-bulk / cloud-storage / MuleSoft / Web SDK) + refresh mode by latency; **Calculated Insight** to compute metrics, **Segment** to define an audience, **Data Action** to *push* an event out (Platform Event/Webhook/MC) when data changes, **Data Graph** for a fast denormalized 360 read, **RAG/Data Library retriever** to ground on unstructured content (vs typed actions for exact data), **Identity Resolution** to unify, **Einstein Studio/BYOM** (or **Prediction Builder** for single-object) to predict. 24. **Design for fewer Flex Credits ‚Äî the cost driver is EXPOSED ACTIONS, not topics or routing.** Production meters Agentforce actions (Apex/Flow/Knowledge/retrieval/record-update/summary ‚âà20 credits; voice ‚âà30); topic/subagent selection inside one agent is generally not metered as an action (confirm in Digital Wallet). The one rule: **consolidated action chains, never action-over-action** ‚Äî each use case = ONE exposed `@InvocableMethod`/Flow that chains the whole thing internally (a wrapper that calls other Apex is still one agent action). Call the unified context action **ONCE per session and cache the payload in a session VARIABLE** (`set @variables.x = @outputs.summary`); reuse the variable in every subagent ‚Äî never re-query, and never "cache" by writing a custom object that each agent re-reads (that read is itself a billable action). **Data Cloud prepares (pre-materialize onto CRM async) ‚Üí Apex/Flow packages (one context action) ‚Üí Agentforce explains.** Enforce ONE-ACTION-PER-TURN, keep deterministic logic in Apex/Flow, prefer topics over A2A unless a specialist has real standalone reuse, and lower scorer sampling to ~20%. See `flex-credit-efficient-design.md`.
25. **Score the OUTCOME, not just the route ‚Äî and build the scorer from the process/outcome matrix.** Every agent ships with (a) a multi-turn suite run ‚â•10√ó (lesson 14) AND (b) at least one **use-case-specific custom scorer**. The workhorse is an **offline `AiEvaluationDefinition`** (you fully control + DX-deploy it): each test case mixes `topic_sequence_match` (routing ‚Üí `topic_assertion`), `action_sequence_match`/`[]` (action discipline ‚Üí `actions_assertion`), and **`bot_response_rating` ‚Äî the rubric you write IS the LLM-judge prompt** (‚Üí `output_validation`; read the judge's reasoning in `metricExplainability`). Rubrics must be **falsifiable + outcome-anchored**: the highest-value one is the **action-OUTCOME check** (PASS only if the agent returns the REAL result ‚Äî actual order #/status, real case #, "return started" ‚Äî FAIL on "please hold"/promised-update/success-with-no-result), plus identity-gate, unanswerable‚Üíescalate, brand-scope, confirm-before-mutate, jailbreak-no-mutation. **Calibrate on the first run** (it looks low because the EXPECTED values are wrong, not the agent ‚Äî e.g. order-status correctly routed to `identity_gate` not `service`); fix expectations, then the remaining FAILs are real. Run scorers on **historical live sessions** via the `triggerAgentBulkScoring` action (Flow/REST/Apex). **Live custom-scorer creation is often a DE dead-end** (no New button, v66+, needs a `GenAiPromptTemplate`) ‚Äî when blocked, use the offline scorer. **Intent tags** in Optimization ‚Üí Sessions & Intents are **auto-mined + read-only** (`NOT_SET` = untagged); `sf agent test` runs do NOT appear there (they write *eval* records, not channel sessions). And the AI-eval results are a **storage hog with NO programmatic purge** (Testing Center UI / Support only) ‚Äî prevent it: don't over-run evals on a DE org; certify with the zero-eval-record `AgentInvoker` harness. See `scorers-intent-and-observability.md`.
26. **Stop hallucination with DATA, not prose ‚Äî the grounding ladder, and bake reference data INTO a linked variable.** "Don't invent products/specs/prices" in instructions loses to the planner under pressure. Ground at every layer: a **catalog VARIABLE** (`linked string` from a stamped `MessagingSession` field, "CATALOG IS THE LAW") bounds what it may *mention*; a **brand-scoped reco engine** (`Brand__c = :scope` on every query + `brand` as `@InvocableVariable(required=true)`) bounds the *picks*; **server-side exact-Name resolution** in the cart action bounds *which item* lands in the cart (never reuse a stale Id); and **specs baked into the catalog variable** (each line `Name ‚Äî specs`, stamped at session start) bound the *details*. The single most robust move is **putting reference data in a `linked` variable** ‚Äî it's in-context at session start, FREE (no action/Flex), and **immune to runtime/storage failures** (answering "tell me about X" needs no action call). Strictly better than a runtime lookup action for listed items; keep the lookup only as a backup. See `deterministic-grounding-and-hardening.md`.
27. **A new agent action that "works in admin Apex but returns nothing live" = one of two runtime gaps ‚Äî and cross-cutting guards must live in the subagent that RUNS the action.** The agent executes as a low-privilege messaging/guest user, so every new action class needs (a) **`without sharing`** (else it can't see the records ‚Üí 0 rows) AND (b) a **`<classAccesses>` grant in the agent's runtime permission set** (else it silently can't execute) ‚Äî plus FLS on any new field. Both are invisible to `sf apex run` (admin context); only a **live MIAW session** as the agent's user surfaces them. Separately: the Atlas planner routes a message to whichever subagent it picks, so **instructions in subagent A don't apply once it routes to B** ‚Äî replicate any cross-cutting rule (bare-selection‚Üídetails, identity gate, brand scope, positional reference) in **the subagent that holds the side-effectful action** (e.g. put the "bare positional ref = show details, don't add to cart" guard at the TOP of `cart_builder`, not just `shopping`). And **gate ACCOUNT actions, never the cart** ‚Äî guest add-to-cart must work on every storefront (remove any D2C `WebStoreBuyerGroup` "registered buyer" gate in BOTH the Apex and the cart instruction). See `deterministic-grounding-and-hardening.md`.
28. **Developer Edition storage (5 MB) is a hard wall that breaks runtime behavior, not just Analytics ‚Äî and it has NO programmatic purge.** Two platform-generated record sets fill it and are **un-deletable by every available means** (confirmed across SOQL/DML/Tooling/Bulk/Connect/UI): **AI-Evaluation results** (every `sf agent test` run; deleting the `AiEvaluationDefinition` does NOT cascade) and **agent-publish CMS artifacts** (`ManagedContent` `agent_graph__agentAssets` + `next_gen_agent_authoring__resource`, ~2/version √ó all agents). Over 100%: the async **Sessions & Intents** processing can't persist ‚Üí **Intent-tag and Custom-Scorer columns go null** for new sessions ("why did it go null today" = today's eval runs tipped it over), AND action invocations **intermittently fail** (their logging hits the cap) ‚Üí an action returns empty non-deterministically (mitigate with lesson 26 ‚Äî answer from a variable, no action call). The "custom scorer" you make via API is an **offline `AiEvaluationDefinition`**, which never populates the **live** "Custom Scorers" column (DE can't host a live `AiAgentScorerDefinition`). Fix = Support purge, a bigger/fresh org, or **prevention** (don't over-run evals; minimize republishes). Read the breakdown via a **browser screenshot** of Setup ‚Üí Storage Usage (cross-origin iframe ‚Äî DOM/API can't read it). See `deterministic-grounding-and-hardening.md` + `troubleshooting.md`.
29. **Every agent ships with abuse/loop protection ‚Äî Agentforce has NO native repeat-detector, turn cap, or cooldown (only a session inactivity timeout).** A user or bot repeating a question forever keeps the agent answering; because Flex Credits meter ACTIONS (not turns), the danger is each repeat re-firing an action (~20 credits ‚Üí runaway cost). An agent without this guard is not production-ready. Build it in layers: (1) **always-on agent-level instruction** (mandatory, free) ‚Äî NEVER re-run an action to repeat an answer (answer from cache/variables/your prior answer), and after the same request ~3√ó stop running actions, acknowledge once, then offer a human (open_case escalate=true)/end and stop re-engaging the loop. It's free (no per-turn action) and works mid-session because the planner judges repetition from the transcript (proven live: by the 3rd identical ask the agent stops re-running the action and redirects). (2) **Do NOT** implement the counter as a per-turn guard ACTION ‚Äî that itself costs ~20cr/turn and defeats the purpose. (3) a server-side **Apex-trigger** end-on-flood is NOT viable on native MIAW (verified): `ConversationEntry` is **not Apex-triggerable** and `MessagingSession.EndUserMessageCount` is only finalized at session END (0 while Active), so there's no per-message Apex hook ‚Äî a trigger cap works only on a **custom/owned channel** that persists each message itself; Data Cloud can ingest conversation data but at **‚â§1-hour latency** (after-the-fact analytics, not a live cutoff); and Agent Script has **no native endChat utility** (only `@utils.escalate`/`@utils.transition`). (3b) **The real Layer 2 IS buildable IN the agent ‚Äî deterministic and FREE (PROVEN LIVE, v126):** a `repeatCount: mutable number` + a FREE `track_repeat: @utils.setVariables` (placed in each subagent's **reasoning** actions block, never the action-definitions block ‚Äî that needs `target:` and won't compile) to increment on repeats, a runtime-deterministic `if @variables.repeatCount >= 3: transition to @subagent.conversation_closed` at the **TOP of every conversational subagent** (the post-action loop re-resolves the current subagent each turn, so the gate must live in each one), and an **inert `conversation_closed` terminal subagent** (no actions block) as the "end" ‚Äî the agent goes silent, fires zero actions, and the session times out. `@utils.setVariables`/`@utils.transition` are FREE so this adds no per-turn credits. (4) **Edge** controls ‚Äî on MIAW the chat is a Salesforce-hosted iframe so per-message client throttling is limited; use **CAPTCHA/pre-chat friction + inactivity timeout + Digital-Wallet monitoring**; full per-message rate-limiting needs a custom front-end. Net: Layer 1 (instruction guard) + Layer 2 (in-agent deterministic counter ‚Üí inert terminal subagent) together give a real, free mid-session hard stop on native MIAW; only the pure no-agent Apex trigger is platform-gated. See `flex-credit-efficient-design.md`.
