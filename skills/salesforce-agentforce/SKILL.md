# Salesforce Agentforce + Data Cloud + Messaging-for-Web Playbook

A practical, **product-agnostic** playbook for building and operating an Agentforce solution on any org, in any industry (commerce, financial services, healthcare, B2B, etc.). The guidance and gotchas are general; concrete examples are drawn from real builds but the patterns apply to any domain — substitute your own objects (an `Order` example maps equally to a Claim, Application, Policy, or Work Order). Follow the DX-first ordering below; only fall to the browser/UI when there is genuinely no API/metadata/CLI path.

## The #0 rule: Official-first — research the supported Salesforce way; never Frankenstein a custom workaround

Before building ANYTHING, find the **official, supported Salesforce mechanism** for the requirement and use it. Research it actively — official Salesforce docs (help.salesforce.com, developer.salesforce.com), the `context7` MCP for product docs, and web search — *based on the actual requirement*, not from memory or assumption. Standard objects and native product features almost always already cover the need.

**Do NOT hand-roll a custom object + Apex to mimic a capability the platform already provides** — not to satisfy a design quickly, not to stay "headless," not to dodge a setup/UI step, not because it's faster to write. That is a **Frankenstein solution**: it demos fine but duplicates a platform feature, bloats storage, drifts from the product's roadmap, reimplements (badly) something Salesforce already does correctly, and becomes permanent tech debt the next person must untangle.

**Real anti-example from this playbook's own history:** a custom `Web_Event__c` object + Apex `IdentityStitchService` was built to do device→person identity stitching and store browsing — when the **official** answer is the **Data Cloud Web SDK + native Identity Resolution** (deviceId/anonymousId match rules). The custom version "worked," but it mirrored high-volume web events into a 5 MB CRM org (repeated storage outages) and re-implemented identity resolution by hand. The right move was to research and use the native Data Cloud capability from the start.

**Decision rule for every requirement:**
1. Ask *"What is the official Salesforce product / feature / standard object / declarative tool for this?"* — then research to confirm; do not guess.
2. Use it in this order: **standard object/native feature → declarative (Flow) → thin Apex glue invoked by Flow → custom object/Apex (last, only with a stated reason no official path exists).**
3. If the official path needs a UI/Setup step (Identity Resolution match rules, a Data Cloud connector, an Einstein Studio model), **do the UI step.** Correct-and-supported beats convenient-and-custom. Never substitute a custom Apex hack just to avoid the official setup.
4. Show the research: when choosing custom, name the official options ruled out and why; when choosing official, name the feature.

"Do it right" = the supported way, even when it's more work or needs a tool you don't currently have — not the fastest custom thing that satisfies the demo. See `crm-vs-datacloud-architecture.md` and `standard-objects-and-declarative-first.md`.

## The #1 rule: DX/CLI/API first, UI last

Preference order for every task:
1. **Salesforce DX / `sf` CLI** — `sf project deploy/retrieve`, `sf data`, `sf apex run`, `sf agent`.
2. **Connect/REST API via CLI** — `sf api request rest "/services/data/vXX.0/..."` (uses the stored session; works even when the raw access token is hidden).
3. **Metadata XML deploy** — author the component, deploy it.
4. **Apex / REST** for anything procedural.
5. **Browser/UI automation** — last resort, and say so first. A handful of things are genuinely UI-only (listed in `troubleshooting.md`): creating CRM data streams, attaching a User-Verification keyset to a MIAW deployment, setting the Person-Account default record type on a profile.

> In a sandboxed agent environment, `sf org display` REDACTS the access token. Do NOT try to scrape it. Instead use `sf api request rest` / `sf data` / `sf apex run`, which all reuse the CLI's stored session under the hood.

## Fast-path checklists

**Deploy a change:**
```bash
sf project deploy start --metadata ApexClass:MyService --target-org MyOrg
sf project deploy report --use-most-recent   # deploys are ASYNC — wait for final "Succeeded"
```
After deploying NEW custom fields, deploy + assign a permission set granting FLS or SOQL will report "No such column".

**Build/ship an agent:**
```bash
sf agent generate agent-spec ... ; sf agent generate authoring-bundle ...
# edit aiAuthoringBundles/<Name>/<Name>.agent
sf agent validate authoring-bundle --api-name <Name>
sf agent publish authoring-bundle --api-name <Name>
sf agent activate --api-name <Name> --version <N>   # ⚠️ publish does NOT activate — new version stays Inactive
# verify the version you published is the ACTIVE one before testing:
sf data query --query "SELECT VersionNumber,Status FROM BotVersion WHERE BotDefinition.DeveloperName='<Name>' AND Status='Active'"
```
Agent type (employee vs Service Agent) is **immutable after first publish** — get it right the first time (see `agentforce-agents.md`).
> ⚠️ **`sf agent publish` does NOT activate the new version** — it's created `Inactive` while the OLD version keeps serving live traffic AND keeps being what `sf agent test`/`run-eval` exercises (they resolve `Status='Active'`). You MUST `sf agent activate --api-name <Name> --version <N>` and then confirm the active version, or you'll "fix" the agent and keep testing the unchanged one. Always verify the active version before drawing ANY conclusion from a test run.

**Run an agent headlessly (from Apex/Flow):**
```apex
AgentInvoker.Response r = AgentInvoker.callAgent('My_Agent_Api_Name', 'user message');
```

**Run a Data Cloud Identity Resolution / Calculated Insight (Connect API):**
```bash
# IR run-now — body is {} and the path is /actions/run-now (NOT /actions/run)
sf api request rest "/services/data/v62.0/ssot/identity-resolutions/<id>/actions/run-now" --method POST --body '{}'
# Calculated Insight run
sf api request rest "/services/data/v62.0/ssot/calculated-insights/<apiName>/actions/run" --method POST --body '{}'
# Ad-hoc SQL query against DLM/DLL/CIO
sf api request rest "/services/data/v62.0/ssot/query-sql" --method POST --body '{"sql":"SELECT COUNT(*) FROM UnifiedssotIndividual__dlm"}'
```

**Diagnose a dead chat / mysterious "can't create records":** check storage FIRST (see `troubleshooting.md`).
```bash
sf api request rest "/services/data/v62.0/limits" | jq '.DataStorageMB'
```

## Top lessons learned (read these even if you read nothing else)

00. **Decide CRM-vs-Data-Cloud placement BEFORE building (the costliest architecture choice).** CRM is the transactional system of record — only the records an agent/Flow/Apex must *act on* (the serviced Account/Contact, Case, in-flight transactional record). Data Cloud is the analytical/unification layer beside it — 360, high-volume events, identity resolution, scores/segments. **Do NOT mirror an entire external system (commerce, ERP, billing, etc.) into CRM to make it agent-readable** — it exhausts CRM row storage, forces brittle Apex single-key matching that forks customers when that key (e.g. email) changes, and leaves a stale 360. Instead: stream-ingest into Data Cloud, let Identity Resolution own matching, Data-Cloud-Triggered-Flow a *thin* record into CRM, and have the agent **read the 360 from the unified profile and write transactional actions to the system of record (CRM or the external API)**. See `crm-vs-datacloud-architecture.md`.

0a. **Discover before you design: interview the user, research the industry, confirm the concrete inputs.** Never build from assumptions about the business process, where data lives, or how identity arrives — ask, and for anything they can't recite, get the artifact (object/field API name, endpoint URL, screenshot of the setting/UI, sample record). Bring researched best-practice as a *proposal* to confirm, not a guess to ship. The build pipeline is **discover → design → build → deploy → verify**. See `discovery-intake.md`.
0. **Design before you prompt: map every process → every outcome → the action that produces it, plus a defined branch for "can't."** An agent built straight into prose instructions, without a capability map + process/outcome matrix + per-process decision tree, WILL hallucinate — it will "trust" an unverified user, narrate actions it never called, and claim refunds/cases that don't exist. Verification is a GATE on a verified variable, never a user's claim ("I'm signed in" ≠ verified). The agent may assert an outcome ONLY if the action returned its success flag this turn; every branch ends in a real result or an honest "I can't" + escalation. See `solution-design-process-outcomes.md`.
0b. **Standard before custom, declarative before code, DX before UI.** Before any custom object/field or Apex class, research whether a STANDARD object already models it (Order/OrderSummary, Case, ReturnOrder, FulfillmentOrder, Shipment, CreditMemo, Account+Contact, Individual/ContactPoint, MessagingSession…) and whether a FLOW can do the automation instead of Apex. Data need: standard object → standard + custom fields → custom object (last). Automation: Flow → thin `@InvocableMethod` Apex called BY a Flow/agent → pure Apex (last). Inspect with `sf sobject describe`; confirm real API names from the Object Reference — never guess `__c`. See `standard-objects-and-declarative-first.md`.
1. **DX/CLI/API beats the screen** for ~90% of Agentforce + Data Cloud work, including things people assume are UI-only (DLO→DMO maps, Calculated Insights, agents, routing flows, smoke tests). Author as metadata + deploy with `sf`; reserve the UI for the handful of genuinely UI-only steps.
2. **Deploys are async** — always confirm final "Succeeded" via `sf project deploy report`, not the first component table.
3. **New custom fields are invisible until a permission set grants FLS** — deploy + assign a permset every time.
4. **Agent type is immutable** after first publish; employee agents must NOT have `default_agent_user` / `connection messaging:` / `@MessagingSession` vars, and a null `BotDefinition.BotUserId` on an employee agent is a reporting artifact, not a failure.
5. **Data Cloud Identity Resolution silently yields 0 unified profiles when the identity keys are sourced from an EMPTY field** — map Individual PK and ContactPointEmail PartyId from a POPULATED field (e.g. `Account_Home__dll.Id__c`), not an empty one like `PersonIndividualId__c`. Verify with `SELECT COUNT(*) FROM Unified...__dlm`.
6. **A Calculated Insight can read a `__dll` (DLO) directly** — you don't always need a DLO→DMO map for analytics CIs.
7. **CRM-source data streams cannot be created via metadata** ("no MktDataTranObject") — that step is UI-gated; everything on top of an existing stream (maps, CIs) is DX-able.
8. **CI SQL has strict rules**: no table aliases (use the full DMO name as qualifier), output aliases must end in `__c`, currency measures must be wrapped in `TRY_CONVERT_CURRENCY(...)`, and a literal target-currency dimension must appear in BOTH SELECT and GROUP BY.
9. **MIAW user verification**: do NOT assume "v2 removes user verification" — modern WebV2 exposes `userVerificationAPI`. The real blocker is the served config's `authMode=UnAuth`; until that's flipped (keyset attached to the deployment, often a Support/UI step) `setIdentityToken` is silently ignored.
10. **MIAW `POST /conversation` → 412 "Conversation Precondition failed" is usually a FULL data-storage org**, not a config error. When storage is exhausted no records (incl. MessagingSession) can be created, so chat, site provisioning, and backfills all fail at once.
11. **AI Evaluation results from `sf agent test` are a massive silent storage hog and are NOT deletable via Apex/Tooling** (Testing Center UI only). Don't run agent tests on a storage-tight org. Deleted records keep consuming storage until you `Database.emptyRecycleBin`.
12. **Agents are conversational, not event-driven** — for self-service automation, fire the agent headlessly from a scheduled job / Flow (via `AgentInvoker`), or do the work in a `without sharing` Apex service. External-webhook-triggered work runs as the locked-down **Site guest user**, so put trusted back-end logic in `without sharing` scheduled sweeps, not guest-context Flows.
13. **Outbound email from a @gmail.com (or any unowned-domain) From address fails Gmail DMARC** even though Apex reports `isSuccess=true`. Real deliverability needs an Org-Wide Email Address on a domain you control with SPF + Salesforce DKIM.
14. **Certify agents with MULTI-TURN conversation tests, repeated to 100% — single-turn tests lie.** An agent that passes single-turn `sf agent test run` will still fabricate actions live, because it behaves differently when users drip info across turns (the normal case). Proven: single-turn 100% green while the same return flow failed ~1-in-3 multi-turn runs. For EVERY agent: document all use cases as a matrix, replay each as a dripped multi-turn conversation via `sf agent test run-eval` (JSON: one `create_session` + a `send_message` per turn + `evaluator.*` assertions), run each critical case ≥10× and require 100% (LLM output is non-deterministic). Never ship a rubric that endorses insecure behavior (e.g. acting on a typed-but-unverified email). See `multi-turn-agent-testing.md`.
15. **"Email limit exceeded" is a sending-METHOD problem, not a provider problem.** The DE ~15/day external-email cap counts only `setToAddresses` sends; emailing a **Lead/Contact/User record via `setTargetObjectId` is cap-EXEMPT** and still sends from your verified OWEA. That's the fix — not a new ESP or a bolted-on relay. (Deliverability — mail "sends" but doesn't arrive — is the separate DMARC/DKIM problem; marketing at scale belongs in Marketing Cloud.) See `email-and-notifications.md`.
16. **Apex agent-action classes: only ONE `@InvocableMethod` per class** (split each action into its own thin wrapper over a shared `static` logic class), inputs/outputs must match the `@InvocableVariable` names, and actions should **return a structured `{success,message}`** (not `void`) so the agent can honestly confirm. And: **you can't edit an Agentforce Builder (Studio) draft headlessly** — rebuild it as a DX `aiAuthoringBundle` with the same dev name and `publish`+`activate` to replace it. See `agentforce-agents.md`.
17. **Agentforce DECIDES; it must not SEND marketing.** Enterprise pattern: agent *nominates* a person into a **Data Cloud segment**, a **consent-aware activation** publishes only consented individuals, and **Marketing Cloud** sends. Unify consent from every source app into Data Cloud's standard Consent model, reconcile to ONE effective consent (latest-wins/opt-out-sticky), and **enforce it at the activation, not in app code.** See `consent-and-marketing-activation.md`.
18. **MIAW identity is bound at conversation creation and never re-checked per turn** — causing a logout PII leak (signed-out user still gets PII) and the mirror over-block (mid-chat login not recognized). Fix the client bridge to **end the conversation on any identity change** (`utilAPI.clearSession({shouldEndSession:true})` + an identity fingerprint) and render identity server-side; the durable fix is **JWT token expiry**. Verified vs guest is **all-or-nothing per deployment** (guests need a separate deployment), and a **cached page/stale nonce** makes a signed-in user look like a guest. See `messaging-web.md`.
19. **Gate privileged actions STRUCTURALLY, not in prose.** Put verification/eligibility in a no-tool "gate" subagent and remove the privileged tools from the pre-auth topic — the planner then physically can't act early. Source every authorization fact from a `linked` channel/record variable (a typed/looked-up value is never proof); require an **opaque OTP proof token** re-verified by Apex on every gated call. See `agent-orchestration-and-memory.md`.
20. **Cross-agent state lives in CRM, not the conversation.** Platform memory doesn't survive a handoff to a different agent or a later headless run — model a per-entity journey object + append-only interaction log + idempotent "done sets" (check-then-mark), and make get-or-create on a unique key race-safe (catch `DUPLICATE_VALUE`, re-SELECT). Make ONE shared-context action the mandatory first call in every subagent. See `agent-orchestration-and-memory.md`.
21. **Integrate with rigor:** verify inbound webhooks by HMAC **over the raw body**; do callouts via Named Credential and **grant the External Credential principal** on the permset; sync two-way by **upsert-by-external-id + reconcile (never skip), per-field ownership, never auto-revoke consent**; compute a **single source-of-truth status** (rank/promote/terminal, evidence-gated, feature-tolerant SOQL); and keep the webhook **guest profile to Apex-class-access only**. See `integration-and-fulfillment.md`.
22. **Know the whole Data Cloud toolbox, pick by the requirement:** ingest by source (CRM connector / Ingestion API streaming-or-bulk / cloud-storage / MuleSoft / Web SDK) + refresh mode by latency; **Calculated Insight** to compute metrics, **Segment** to define an audience, **Data Action** to *push* an event out (Platform Event/Webhook/MC) when data changes, **Data Graph** for a fast denormalized 360 read, **RAG/Data Library retriever** to ground on unstructured content (vs typed actions for exact data), **Identity Resolution** to unify, **Einstein Studio/BYOM** (or **Prediction Builder** for single-object) to predict. See `data-cloud-ingestion-actions-grounding.md` + `data-cloud.md`.
